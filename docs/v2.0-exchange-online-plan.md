# v2.0.0 ‚Äî Exchange Online integration plan

Status: **in progress** (Phase 0 spike complete ‚úÖ ‚Äî see [v2.0-spike-findings.md](v2.0-spike-findings.md)). Work happens on branch `v2.0.0`.

## Goal
Add Exchange Online mailbox management so a departing user's mailbox survives unlicensing ‚Äî primarily as steps in the termination scenario, and as ad-hoc actions on a mailbox.

## Locked decisions
- **Tenant is pure-cloud** ‚Äî NO hybrid Exchange. Operate directly in Exchange Online; the `Set-RemoteMailbox`/dirsync-revert problem does not apply.
- **Channel: out-of-process `pwsh` (REVISED 2026-07-08).** Run the `ExchangeOnlineManagement` module in a separate `pwsh` process (persistent session, stdin/stdout JSON line protocol; the EXO token is borrowed from the Entra sign-in and passed via stdin). Everything sits behind an `IExchangeService` interface, so this swap was contained to `ExchangeService`.
  - **Originally** we chose in-process hosting (`Microsoft.PowerShell.SDK` + the module in a runspace). That was **abandoned after the first live test**: the EXO module bundles MSAL 4.74 / Azure.Core 1.44, which clash with the Graph SDK's already-loaded MSAL 4.83 / Azure.Core 1.53 in a single process ‚Üí `Connect-ExchangeOnline` fails with "Module could not be correctly formed." A separate process has clean assemblies + a correct module env, fixing the whole class of in-proc problems. Bonus: the heavy `Microsoft.PowerShell.SDK` dependency is dropped.
- **Features:**
  - Convert mailbox **Regular ‚Üî Shared only** (no Room/Equipment).
  - **Internal-only forwarding** via `Set-Mailbox -ForwardingAddress <internal recipient>` (+ `DeliverToMailboxAndForward`). External `ForwardingSmtpAddress` is out of scope.
  - Read mailbox state (type + forwarding) for display and guardrails.
- **Identity = UPN**, but the UI must also support **lookup/search by name** ‚Äî both for picking the subject mailbox and the internal forwarding target (a mail-enabled recipient: user / shared mailbox / distribution group).

## Core use case (ordering matters)
Termination flow: **disable AD ‚Üí convert mailbox to Shared ‚Üí set forwarding ‚Üí remove license.** Guardrail: before removing a license, check mailbox type and warn/block if not yet Shared (a regular mailbox loses data when unlicensed; a shared mailbox is safe, 50 GB cap unlicensed).

## Why this isn't "just more Graph"
Mailbox convert + forwarding are **Exchange-only** (`Set-Mailbox`); they are not in Microsoft Graph. Exchange admin lives behind a different resource (`outlook.office365.com`) with its own delegated permission and RBAC roles.

## Auth design (verified in the spike)
- Reuse the existing `InteractiveBrowserCredential` (in `GraphService`) to mint an Exchange token silently: scope `https://outlook.office365.com/.default`. No second browser sign-in.
- `Connect-ExchangeOnline -AccessToken <token> -Organization <tenant> -ShowBanner:$false -SkipLoadingFormatData`.

## Tenant prerequisites (done once, in Entra ‚Äî cannot be automated from the app)
1. ‚úÖ REQUIRED ‚Äî delegated **`Exchange.Manage`** (Office 365 Exchange Online) granted to the app registration + admin consent. **This is the one the ExchangeOnlineManagement module's `-AccessToken` flow needs** ‚Äî confirmed live 2026-07-08 (token with only `Exchange.ManageV2` failed to form the module; adding `Exchange.Manage` fixed it). Note: `Exchange.ManageV2` is for the REST Admin API only (which can't convert mailboxes / set forwarding) ‚Äî it is NOT sufficient for this app. After granting, sign out/in of Entra in the app so the token is reissued with the new scope.
2. ‚úÖ DONE ‚Äî the signing-in admin holds an Exchange RBAC role (Recipient Management / Exchange Administrator), confirmed 2026-07-07.

**Both prerequisites met ‚Üí the read path is validated live (2026-07-08): connect + Get-Mailbox works end-to-end.** Write actions (convert / forwarding) still to be exercised live.

### Runtime prerequisite (out-of-process channel)
Each admin machine must have **PowerShell 7 (`pwsh`)** + the **`ExchangeOnlineManagement`** module installed (both present on the dev box: pwsh 7, EXO 3.9.2). The app locates `pwsh` on PATH or at `%ProgramFiles%\PowerShell\7\pwsh.exe`. Phase 4 deployment must document/enforce this (or bundle a pwsh + the module). The app no longer bundles any PowerShell itself.

## Build map (mirrors existing Graph patterns)
- `Models/MailboxType.cs`, `Models/MailboxInfo.cs`, `Models/MailboxRecipient.cs`.
- `Services/IExchangeService.cs` + `ExchangeService.cs` ‚Äî connect lifecycle, `GetMailboxAsync`, `ConvertMailboxAsync`, `SetForwardingAsync`/`ClearForwardingAsync`, `SearchRecipientsAsync` (forwarding-target picker).
- `Services/ExchangeErrors.cs` ‚Äî map cmdlet ErrorRecords to friendly messages (mirrors `GraphErrors.cs`).
- `IGraphService.GetAccessTokenAsync(scopes)` ‚Äî lets sibling services borrow a token from the signed-in credential.
- Single `Runspace`, reused; all cmdlet calls serialized behind a `SemaphoreSlim` (like `_skuGate`); run off the UI thread. Static access token expires ~1h ‚Üí re-acquire silently and reconnect on auth failure.
- Scenario: add to `ScenarioActionType` ‚Äî `ExchangeConvertToShared`, `ExchangeConvertToRegular`, `ExchangeSetForwarding`, `ExchangeClearForwarding`. Add a `DeliverAndForward` bool (+ forwarding-target ref) to `ScenarioStep`. Wire into `ScenarioRunner` beside the `Cloud*` cases; feed `SaveOperationLog`.
- UI: scenario step editor entries; interactive Mailbox section on the cloud user detail pane; EXO connection status; confirm-before-write.
- Installer: `Product.wxs` + `publish.ps1` include PS SDK assemblies + bundle the `ExchangeOnlineManagement` module folder (import by path; do not rely on `Install-Module` at runtime). Expect a large size increase.

### Phase 2 notes
- Actions added: `ExchangeConvertToShared/Regular`, `ExchangeSetForwarding`, `ExchangeClearForwarding`. `ScenarioStep` gained `ForwardingTarget` (nullable) + `DeliverAndForward`. Runner resolves the mailbox identity from the **live per-step DN** (survives an earlier `MoveToOu`); this also fixed the same latent bug in the existing cloud resolvers.
- Editor groundwork done: `ShowForwarding` flag + `ForwardingTarget`/`DeliverAndForward` carried losslessly through the row‚Üîstep round-trip. **Phase 3** still owes the actual forwarding-target picker UI (a recipient search dialog backed by `IExchangeService.SearchRecipientsAsync`) + the XAML panel gated on `ShowForwarding`.
- Deferred hardening (Phase 4): the hosted EXO PowerShell pipeline has **no timeout/cancellation backstop** (the scenario progress window passes `CancellationToken.None` and has no Cancel button). A hung EXO connect/cmdlet would block a run. Pre-existing pattern for cloud steps, but worse for EXO (no HttpClient timeout). Add a linked-CTS timeout in `ExchangeService` and/or a Cancel button.
