# v2.0.0 — Exchange Online integration plan

Status: **in progress** (Phase 0 spike complete ✅ — see [v2.0-spike-findings.md](v2.0-spike-findings.md)). Work happens on branch `v2.0.0`.

## Goal
Add Exchange Online mailbox management so a departing user's mailbox survives unlicensing — primarily as steps in the termination scenario, and as ad-hoc actions on a mailbox.

## Locked decisions
- **Tenant is pure-cloud** — NO hybrid Exchange. Operate directly in Exchange Online; the `Set-RemoteMailbox`/dirsync-revert problem does not apply.
- **Channel: hosted PowerShell module.** Bundle `Microsoft.PowerShell.SDK` + the `ExchangeOnlineManagement` module and run cmdlets in-process via `Connect-ExchangeOnline -AccessToken`. (Chosen over the Preview REST Admin API and over shelling out to `pwsh`.) Everything sits behind an `IExchangeService` interface so the channel can be swapped later.
- **Features:**
  - Convert mailbox **Regular ↔ Shared only** (no Room/Equipment).
  - **Internal-only forwarding** via `Set-Mailbox -ForwardingAddress <internal recipient>` (+ `DeliverToMailboxAndForward`). External `ForwardingSmtpAddress` is out of scope.
  - Read mailbox state (type + forwarding) for display and guardrails.
- **Identity = UPN**, but the UI must also support **lookup/search by name** — both for picking the subject mailbox and the internal forwarding target (a mail-enabled recipient: user / shared mailbox / distribution group).

## Core use case (ordering matters)
Termination flow: **disable AD → convert mailbox to Shared → set forwarding → remove license.** Guardrail: before removing a license, check mailbox type and warn/block if not yet Shared (a regular mailbox loses data when unlicensed; a shared mailbox is safe, 50 GB cap unlicensed).

## Why this isn't "just more Graph"
Mailbox convert + forwarding are **Exchange-only** (`Set-Mailbox`); they are not in Microsoft Graph. Exchange admin lives behind a different resource (`outlook.office365.com`) with its own delegated permission and RBAC roles.

## Auth design (verified in the spike)
- Reuse the existing `InteractiveBrowserCredential` (in `GraphService`) to mint an Exchange token silently: scope `https://outlook.office365.com/.default`. No second browser sign-in.
- `Connect-ExchangeOnline -AccessToken <token> -Organization <tenant> -ShowBanner:$false -SkipLoadingFormatData`.

## Build map (mirrors existing Graph patterns)
- `Models/MailboxType.cs`, `Models/MailboxInfo.cs`, `Models/MailboxRecipient.cs`.
- `Services/IExchangeService.cs` + `ExchangeService.cs` — connect lifecycle, `GetMailboxAsync`, `ConvertMailboxAsync`, `SetForwardingAsync`/`ClearForwardingAsync`, `SearchRecipientsAsync` (forwarding-target picker).
- `Services/ExchangeErrors.cs` — map cmdlet ErrorRecords to friendly messages (mirrors `GraphErrors.cs`).
- `IGraphService.GetAccessTokenAsync(scopes)` — lets sibling services borrow a token from the signed-in credential.
- Single `Runspace`, reused; all cmdlet calls serialized behind a `SemaphoreSlim` (like `_skuGate`); run off the UI thread. Static access token expires ~1h → re-acquire silently and reconnect on auth failure.
- Scenario: add to `ScenarioActionType` — `ExchangeConvertToShared`, `ExchangeConvertToRegular`, `ExchangeSetForwarding`, `ExchangeClearForwarding`. Add a `DeliverAndForward` bool (+ forwarding-target ref) to `ScenarioStep`. Wire into `ScenarioRunner` beside the `Cloud*` cases; feed `SaveOperationLog`.
- UI: scenario step editor entries; interactive Mailbox section on the cloud user detail pane; EXO connection status; confirm-before-write.
- Installer: `Product.wxs` + `publish.ps1` include PS SDK assemblies + bundle the `ExchangeOnlineManagement` module folder (import by path; do not rely on `Install-Module` at runtime). Expect a large size increase.

### Phase 2 notes
- Actions added: `ExchangeConvertToShared/Regular`, `ExchangeSetForwarding`, `ExchangeClearForwarding`. `ScenarioStep` gained `ForwardingTarget` (nullable) + `DeliverAndForward`. Runner resolves the mailbox identity from the **live per-step DN** (survives an earlier `MoveToOu`); this also fixed the same latent bug in the existing cloud resolvers.
- Editor groundwork done: `ShowForwarding` flag + `ForwardingTarget`/`DeliverAndForward` carried losslessly through the row↔step round-trip. **Phase 3** still owes the actual forwarding-target picker UI (a recipient search dialog backed by `IExchangeService.SearchRecipientsAsync`) + the XAML panel gated on `ShowForwarding`.
- Deferred hardening (Phase 4): the hosted EXO PowerShell pipeline has **no timeout/cancellation backstop** (the scenario progress window passes `CancellationToken.None` and has no Cancel button). A hung EXO connect/cmdlet would block a run. Pre-existing pattern for cloud steps, but worse for EXO (no HttpClient timeout). Add a linked-CTS timeout in `ExchangeService` and/or a Cancel button.
