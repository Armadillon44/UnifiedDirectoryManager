# v2.0.0 — Phase 0 spike findings (hosted PowerShell + EXO module)

Ran on the build machine (.NET SDK 10.0.301, pwsh 7, ExchangeOnlineManagement 3.9.2). A throwaway console harness referenced the app's conflict surface (Microsoft.Graph 6.2.0 + Azure.Identity 1.21.0 + Microsoft.PowerShell.SDK 7.6.3, `net10.0-windows` + `UseWPF`).

## Proven (green)
- **net10 PowerShell SDK exists**: `Microsoft.PowerShell.SDK` 7.6.0 GA (tested 7.6.3) targets .NET 10. Added to the real `.csproj` → `dotnet restore` + `dotnet build` succeeded with **0 warnings, 0 errors**.
- **In-proc hosting works**: ExchangeOnlineManagement 3.9.2 imports cleanly; `Connect-ExchangeOnline` resolves.
- **Auth design confirmed**: `Connect-ExchangeOnline` exposes both **`-AccessToken`** and **`-SkipLoadingFormatData`** (verified via reflection). So we can mint an Exchange token from the existing `InteractiveBrowserCredential` and pass it in — no second sign-in.
- **No assembly double-load**: only single versions loaded (Azure.Core/Azure.Identity assembly stamp 1.53.0.0, Microsoft.Identity.Client 4.83, System.Management.Automation 7.6.0.500). No multiple-version conflicts.
- **No NuGet package conflict.** Azure.Identity stays on package **1.21.0** (the latest published — there is no 1.53 package). The `1.53.0.0` seen at runtime is the Azure SDK *assembly version*, which is decoupled from the package version. The app keeps `Azure.Identity` at 1.21.0; restore + build are clean (0/0) with the PS SDK alongside.

## Required patterns / gotchas (baked into ExchangeService)
- **Runspace must be configured deliberately.** A default runspace failed with `AuthorizationManager check failed` loading format ps1xml — fixed by `iss.ExecutionPolicy = ExecutionPolicy.Bypass` (that's what silences it) plus `-SkipLoadingFormatData` on connect. **Use `InitialSessionState.CreateDefault()` (full), NOT `CreateDefault2()`.** CORRECTION (found during the first live test): the spike credited `CreateDefault2` because it only tested *Import-Module*, not a live *connect*. `Connect-ExchangeOnline` downloads a cloud-side "WebModule" of the mailbox cmdlets and forms it using the standard engine modules + PackageManagement/PowerShellGet; the minimal CreateDefault2 session starves that step and the connect fails with "Module could not be correctly formed. Please run Connect-ExchangeOnline again." CreateDefault() + Bypass works for both import and connect.
- **Runtime coexistence smoke test still owed**: the spike constructed `InteractiveBrowserCredential` and hosted PowerShell in one process without issue, but do a manual Graph interactive sign-in once with the PS SDK referenced to confirm nothing regresses at runtime.
- **`Set-Mailbox` / `Get-Mailbox` don't exist until after `Connect-ExchangeOnline`** (EXO v3 loads commands on connect). Don't probe for them pre-connect.
- Restore emits a benign NU1603 (PS SDK wants `System.DirectoryServices.Protocols` >= 10.0.5; app pins 10.0.9). Harmless.

## Live-test outcome (2026-07-08): in-process hosting abandoned → out-of-process pwsh
The first live connect failed with **"Module could not be correctly formed. Please run Connect-ExchangeOnline again."** Root cause: an **in-process assembly-version conflict**. When the admin signs into Entra, the Graph SDK loads MSAL **4.83** / Azure.Core **1.53**; `Connect-ExchangeOnline` then runs in the same process but the EXO 3.9.2 module bundles/expects MSAL **4.74** / Azure.Core **1.44**, so it gets the wrong (already-loaded, newer) versions and its cloud "WebModule" can't be formed. `CreateDefault()` did NOT fix it (the earlier "CreateDefault2 is the recipe" conclusion was also wrong — see above).

**Resolution:** dropped `Microsoft.PowerShell.SDK` and the in-process runspace entirely; `ExchangeService` now runs the module in a separate `pwsh` process (persistent session, stdin/stdout JSON line protocol; token passed via stdin). Verified out-of-process: the host imports the module and `Connect-ExchangeOnline` runs cleanly — a bogus token returns a framed `{"ok":false,"error":"UnAuthorized"}` (i.e. the module forms fine; only auth is rejected), which is exactly the behavior a real token needs. Lesson: this whole in-proc saga is why "shell out to pwsh" is the robust channel for EXO.
